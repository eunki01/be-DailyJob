# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 트랜잭션 추가 및 계층 분리를 위한 다이나믹 프록시 도입

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 데이터베이스 클래스에서 모든 테이블과 키에 대한 접근을 하는 대신에 데이터베이스가 커넥션 생성과 관리, 실행에 관한 책임을 지니도록 함. 이 과정에서 다이나믹 프록시를 도입하고 데이터베이스가 쿼리 생성하는 클래스에 의존하는 게 아니라 쿼리를 생성하는 클래스가 데이터베이스에 의존하도록 의존 관계를 변경함.
- 그렇게 판단한 이유: 현재 추상화된 구조는 비즈니스 계층에서 db 접근에 대한 부분이 데이터베이스로 가려져 있긴 하지만 테이블과 키를 지정해줘야 하기 때문에 강하게 결합된 구조라고 볼 수 있음. 이에 더해 트랜잭션을 걸기 위해서는 결국 비즈니스 로직이 커넥션 사이에 존재해야 하기 때문에 계층 분리가 어려움. 이를 해결하기 위한 방법이 다이나믹 프록시라고 생각해서 이를 적용함.
쿼리 생성에 대한 클래스와 커넥션 생성과 실행에 대한 클래스의 의존관계를 바꾼 이유는 다음과 같음.
데이터베이스 -> 레포지토리로 의존 관계가 있으면 레포지토리 내용을 데이터베이스가 알아야 하기 때문에 새로운 레포지토리가 생성되면 데이터베이스 객체도 수정되어야 함.
레포지토리 -> 데이터베이스로 의존 관계가 있으면 레포지토리는 범용적인 기능을 제공하는 클래스인 데이터베이스를 사용하게 되고 데이터베이스 자체는 레포지토리가 추가된다고 해도 변경사항이 존재하지 않게 됨.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 비즈니스 로직엔 데이터베이스에 관한 부분이 드러나지 않아 가독성이 좋아짐. 또한 세부사항을 가진 클래스가 범용적인 기능을 하는 클래스에 의존하도록 의존관계를 변경함에 따라 유지보수성이 좋아짐. 지원하지 못했던 트랜잭션 처리에 대해서 지원할 수 있게 되었고 다양한 쿼리들도 사용할 수 있게 됨.

### 3. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 마지막 날인 만큼 ai를 적극적으로 쓴 것.
- 그 접근이 도움이 된 이유: 그 동안 생각했던 것들을 적용해 보고 코드를 살펴보면서 당위성에 대해서 생각해 볼 수 있었음. 의존관계를 제대로 잡아야 유지보수성이 좋아질 수 있다는 것을 알게 되었고 지금까지 내가 의존관계에 대해서 고민을 제대로 하지 않았다는 점을 알아갈 수 있었던 것 같음.
- 다음에도 반복하고 싶은 점: ai에 전체를 맡기진 않더라도 ai를 썼을 때 배워갈 수 있는 부분이 있다는 생각이 들었고 남은 기간 동안 잘 사용하고 싶음.

### 4. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 스쿼드 세션을 할 때 괜찮은 고민이나 질문을 못 한 것.
- 그렇게 된 원인: 팀 세션을 한번 했기 때문에 코드 레벨에서 살펴보지 않았던 것, 구현에 시간을 쓰고 정작 구조적인 고민이나 기술적인 부분에 대해서 공부를 못한 점.
- 다음에는 다르게 해볼 구체적인 행동: 구현을 어느 정도는 내려놓을 것.

### 5. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
데이터베이스 같은 경우 어설프게 추상화하려다 보니 테이블 별로 필요한 쿼리를 정의하고 쓰는 게 어렵고 타겟 테이블이랑 키를 넘겨줘야 되는 구조잖아. 이건 잘못됐다고 보는데 어때? 그리고 트랜잭션 처리를 위해서는 database는 직접 실행보다는 커넥션 관리에 대한 책임을 갖는 게 맞는 것 같아서 이 방향성으로 수정해 줘. 데이터베이스 계층 분리를 위해서 다이나믹 프록시 추가해 줘. 데이터베이스 수정하면 조인 쿼리도 쓸 수 있으니까 메인 페이지 조회는 조인 쿼리로 처리하고 article 작성이랑 mypage 수정도 트랜잭션이 필요하니까 추가해 줘.

선택한 이유: 그래도 방향성에 대해서 제시했다고 생각함.

### 6. 원본 프롬프트 구조 분석
- 문제 상황: 어설픈 데이터베이스 추상화로 인해 테이블별 필수 쿼리 정의 및 사용이 어려움.
           쿼리를 실행할 때 타겟 테이블과 키를 매번 직접 넘겨주어야 하는 비효율적인 구조를 가지고 있음.
- 원하는 결과를 어떻게 요구했는가: 기존 추상화 방식의 문제점을 지적하며, 올바른 방향성인지에 대한 피드백을 구함.
                           구체적인 리팩토링 요구사항을 제시

                           메인 페이지 조회를 위한 조인(Join) 쿼리 구현
                           article 작성 및 mypage 수정 기능에 트랜잭션(Transaction) 처리 적용

- 판단 기준이나 제약 조건이 있는가: 데이터베이스 계층 분리와 '트랜잭션 일관성 보장이 핵심 판단 기준임.
                            직접적인 DB 실행을 막고 커넥션을 관리하는 구조로 변경해야 한다는 제약 조건이 있음.
- 출력 방식이 명확한가: X
- 나의 가설, 우려, 의도가 드러나는가: 데이터베이스 계층이 쿼리 실행을 직접 담당하기보다 커넥션/트랜잭션 제어에 집중하고, 비즈니스 로직과의 분리를 위해 프록시 패턴을 쓰는 것이 아키텍처적으로 더 우수할 것이라는 확신이 보임. 어설픈 추상화(테이블/키를 직접 넘기는 방식)가 결합도를 높이고 생산성을 떨어뜨리고 있다는 점에 대해 강한 문제의식을 느낌.

### 7. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
[시스템 및 기술 스택 컨텍스트]
언어 및 환경: Java 17 / 순수 JDBC (Spring Boot 사용 안 함 / 프레임워크 없이 직접 구현 중)
디자인 패턴: Dynamic Proxy를 이용한 트랜잭션 관리, Repository 패턴 기반의 데이터베이스 계층 분리
[질문 및 문제 상황]
현재 프로젝트에서 데이터베이스 계층을 어설프게 추상화하다 보니, 다음과 같은 한계에 부딪혔습니다.
	1.	특정 테이블에 종속적인 쿼리를 유연하게 정의하기 어렵습니다.
	2.	데이터베이스 접근 객체에 타겟 테이블과 키(Key)를 파라미터로 매번 넘겨주어야 하는 강한 결합도 문제가 발생합니다.
이 구조가 잘못되었다는 제 판단에 대해 아키텍처적 관점에서 피드백을 주세요.
[개선 및 제약 조건]
위 문제를 해결하기 위해 아래 구조로 리팩토링을 진행하고자 합니다. 다음 제약 조건을 만족하는 가이드와 코드를 작성해 주세요.
	1.	책임의 분리 (Separation of Concerns): Database 클래스는 쿼리를 직접 실행하지 않고, **커넥션 획득 및 반납(Connection Management)**과 트랜잭션 관리의 책임만 갖도록 수정해 주세요.
	2.	계층 분리 (Dynamic Proxy): 비즈니스 로직(Service 계층)과 데이터베이스 접근 기술이 섞이지 않도록, **JDK Dynamic Proxy(또는 CGLIB)**를 사용하여 @Transactional 처럼 동작하는 트랜잭션 프록시 레이어를 추가해 주세요. (ThreadLocal을 활용한 커넥션 동기화 방식 선호)
	3.	조인(Join) 쿼리 구현: 데이터베이스 추상화 장벽을 낮춰 직접 SQL 조인을 쓸 수 있게 됨에 따라, 메인 페이지 조회(예: Article과 User 정보를 조인하여 조회) 기능을 직접 SQL로 구현해 주세요.
	4.	트랜잭션 보장: article 작성과 mypage 수정 기능은 데이터 정합성이 중요하므로, 프록시를 통해 하나의 트랜잭션(커넥션) 내에서 원자적(Atomic)으로 처리되도록 비즈니스 로직과 트랜잭션 경계 설정 코드를 보여주세요.
[출력 형식]
	1.	아키텍처 설계 구조도: 요청 흐름(Service -> Proxy -> Repository -> Database (Connection))을 텍스트 기반 시퀀스 다이어그램이나 간단한 다이어그램으로 도식화해 주세요.
	2.	리팩토링 코드 세트:
        TransactionProxy (다이나믹 프록시 구현체)
        ConnectionManager 또는 수정된 Database 클래스
        ArticleService (트랜잭션이 적용된 비즈니스 로직 예시)
        ArticleRepository (조인 쿼리가 적용된 데이터 접근 예시)
	3.	핵심 설명: 이전 구조 대비 이 방식이 갖는 결합도 해소 측면의 장점을 3가지로 요약해 주세요.

### 8. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등
할루시네이션(환각) 방지 및 기술 일치:
기존 프롬프트는 어떤 언어로 작성해야 하는지 유추해야 하므로, AI가 엉뚱한 언어(예: Python, Node.js)로 짜거나 Spring Boot의 @Transactional을 띡 던져주고 끝낼 위험이 큽니다. 기술 스택을 명시함으로써 본인이 작업 중인 환경에 즉시 복사-붙여넣기 할 수 있는 코드가 출력됩니다.

동작 가능한 '엔드투엔드(E2E)' 코드 확보:
프록시, 커넥션 풀 관리, 서비스, 리포지토리가 각각 어떻게 유기적으로 맞물리는지 구체적인 클래스 리스트를 지정해 주었기 때문에, 개별 개념 설명에 그치지 않고 전체 흐름을 관통하는 완성도 높은 설계 템플릿을 얻을 수 있습니다.

가독성 극대화:
출력 형식을 구조도와 코드 세트, 3줄 요약으로 명확히 끊어달라고 지정했기 때문에, 장황한 설명글 대신 가독성이 높고 핵심만 짚은 깔끔한 답변을 받아볼 수 있습니다.
