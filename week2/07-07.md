# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 로그인 시 쿠키 발급 기능, HttpResponse 생성에 대한 처리

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Builder와 정적 팩소리 메소드로 HttpResponse를 생성하는 것을 처리한 것. 
- 그렇게 판단한 이유: HttpResponse에 들어갈 수 있는 헤더 종류가 많고 모든 헤더가 필요한 것은 아니므로 응답을 생성할 때 필요한 부분만 골라넣을 수 있다는 점에서 Builder 패턴을 선택함. 이에 더해 별도로 헤더를 추가하지 않는 경우에 사용할 기본 에러 Response 생성을 정적 팩소리 메소드로 만들고 HttpResponseFactory로 모으면 새로운 Response를 추가할 때 HttpResponseFactory만 보면 돼서 편리할 거라 생각함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 새로운 HttpResponse가 필요하다고 하더라도 Builder와 HttpResponseFactory에만 추가하고 사용하면 된다는 점에서 편의성이 올라가고 응집도가 올라간 것 같음.

### 3. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 디자인 패턴들을 계속 보면서 적용할 수 있을만한 부분을 찾아본 것
- 그 접근이 도움이 된 이유: 생성, 구조, 행위로 나눠진 만큼 큰 틀에서 기능을 생성, 구조, 행위로 나누면 되겠다는 아이디어도 얻을 수 있었고 적용할 수 있을만한 부분들을 찾는 데 도움이 된 것 같음.
- 다음에도 반복하고 싶은 점: 계속 살펴보면서 클래스 다이어그램 표기법에 익숙해지고 디자인 패턴을 정확하게 적용하고 싶음.

### 4. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 디자인 패턴을 적용하겠다고 너무 많은 시간을 쓴 점. RequestHandler 내부에서 MethodHandler를 호출하는 부분이 있는데 이 부분이 애초에 클래스가 아니기 때문에 proxy를 가져다 쓰는 게 아니라 그냥 클래스를 만들어서 필요한 경우 인증을 하고 MethodHandler로 접근하도록 했어야 했음.
- 그렇게 된 원인: proxy에 대한 이해가 부족했고 다형성의 목적을 잊은 것.
- 다음에는 다르게 해볼 구체적인 행동: 개념을 확실하게 이해하고 정신을 차려야 함.
