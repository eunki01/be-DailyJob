# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹 서버 4단계 - POST로 회원가입

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Responser에 path 이외에도 상태 메세지도 전달하도록 한 것, DataOutputStream을 OutputStream으로 바꾼 것.
- 그렇게 판단한 이유: 핸들러에서 발생한 에러에 대한 상태값을 전달하기 위해 path를 사용하는 것은 좋지 않다고 생각했기 때문임. OutputStream만으로도 전송이 가능함.
- 고려했지만 선택하지 않은 대안: Responser 내부에서 상태값을 정하도록 하려고 했으나 파일 접근에 대한 오류는 responser 내부에서만 일어나지만 methodhandler에서 발생할 수 있는 에러에 대한 응답을 만드는 게 애매해졌음.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 외부에서 상태값을 정해주기 떄문에 어떤 응답이 발생할지 명확해짐.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: http body를 BufferedReader의 readline으로 읽으니 이후로 아무것도 동작하지 않았음.
- 처음 추측한 원인: FE에서 데이터가 제대로 전송되지 않았다고 생각했음.
- 원인을 확인하기 위해 시도한 방법: form data를 쏘면서 로그를 남기고 브라우저 개발자 도구로 네트워크로 나가는 패킷을 확인함. 서버 측에서는 전후 로그를 찍어서 어디까지 프로그램이 진행되었는지 확인함.
- 실제 원인: BufferedReader의 경우 readline으로 읽을 때 EOF나 개행 문자를 만날 때까지 계속 대기하게 되고 body 뒤에는 개행 문자가 있으리라는 보장이 없기 때문에 발생한 문제임.
- 해결 방법: Http header의 content-Length만큼 read함수를 통해 읽어서 버퍼에 저장함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 로그가 찍히는 위치를 확인하고 별도 에러가 없으면 스레드가 대기 중인 것을 알아차릴 것. 구체적인 메소드의 동작에 대해서 찾아볼 것.

### 4. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 디버깅을 할 때 콘솔로 찍을 게 아니라 네트워크 패킷을 바로 확인했어야 했고 서버에서도 로그를 여러 구간에 찍었어야 함.
- 그렇게 된 원인: 디버깅에 익숙하지 않음.
- 다음에는 다르게 해볼 구체적인 행동: 네트워크 탭을 먼저 확인하고 로그를 수시로 찍어볼 것.
