- 그렇게 판단한 이유: Enum으로 Mediatype, httpStatus을 정의한 이유는 객체가 하나만 생성되므로 성능에 있어서 좋고 typecheck를 통해 content-type 헤더에 들어갈 문자열을 enum 클래스에 추상화시켜서 url만 전달하여 타입을 얻을 수 있게 되고 타입 체크 안정성이 올라가는 점을 고려해서 진행함.
UrlInfo의 경우에는 contentType을 하드코딩해서 넣는 부분을 제거하기 위한 것과 be-was 1주차 3단계에서 registration 디렉토리로 접근할 때 index.html로 접근하도록 해 줘야 하는 부분이 있었는데 모든 디렉토리 접근 시 index.html이 존재한다면 index.html을 반환하고 없다면 NOT_FOUND로 처리하기 위해서 필요한 정보라고 생각함.
UrlDecoder의 경우에는 UrlDecoder의 경우 urlencoding과 query 추출을 고려해서 추가하게 되었음. 사실 requestParser에 같이 들어가도 되는 기능이라는 생각도 했었으나 query가 없다면 사용하지 않게 되는 메소드이므로 나누는 게 맞다고 보았음.

### 6. 원본 프롬프트 구조 분석
- 문제 상황: http 문자열 파싱 시 String constant pool에 등록하는 방안에 대한 평가 요구
- 원하는 결과를 어떻게 요구했는가: 없음
- 필요한 맥락이 포함되어 있는가: http 헤더 구성 요소의 재사용성 언급
- 판단 기준이나 제약 조건이 있는가: 없음
- 출력 방식이 명확한가: 없음
- 나의 가설, 우려, 의도가 드러나는가: Http 헤더를 구성요소를 string constant pool 등록하면 String 객체가 덜 생성되고 이는 major gc 발생 빈도를 낮춰서 성능이 향상될 것이라고 생각했으나 이 부분을 적진 않았음.

- 프롬프트 개선안

	2.	악의적인 요청(예: 무작위 헤더를 대량으로 보내는 공격)으로 인해 String Pool이 비대해져 메모리 누수(OOM)나 성능 저하가 발생할 위험은 어떻게 제어해야 할까요?

### 8. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등
```
1. 컨텍스트의 명확화 (고성능, 대량 요청 환경)
HTTP 파싱 최적화를 고민한다는 것 자체가 트래픽이 몰리는 환경을 전제로 합니다. 질문에 '고성능 서버 환경'이라는 기준을 추가하면, AI나 전문가가 단순히 "네, 좋은 생각입니다"를 넘어 힙 메모리와 GC(Garbage Collection) 성능까지 고려한 답변을 줍니다.
2. 현실적인 리스크(보안 및 OOM) 언급
HTTP 요청은 외부 사용자가 보냅니다. 해커가 X-Random-Header-1: abc1, X-Random-Header-2: abc2처럼 매번 다른 임의의 헤더를 수백만 개 보내면, String Pool이 가득 차서 서버가 뻗을 수 있습니다(정적 문자열이 아니기 때문). 이 리스크를 제어하는 방법을 물어봄으로써 답변의 질을 높입니다.
3. 실무 모범 사례(Best Practice) 유도
톰캣(Tomcat)이나 네티(Netty) 같은 프레임워크들은 이미 이 고민을 끝내고 자신들만의 방식으로 구현해 두었습니다. (참고로 이들은 String Pool 대신, 자주 쓰이는 헤더들을 미리 byte[]나 static 상수로 정의해 두고 매핑하는 자체 캐싱 방식을 주로 씁니다.) 이를 언급하면 오픈소스 분석 기반의 깊이 있는 답변을 얻을 수 있습니다.
```
