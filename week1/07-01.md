# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- InputStream, InputStreamReader, BufferedReader 차이점 학습
- 스레드풀의 필요성과 TreadPoolExecutor 각 매개변수의 의미 학습
- socket programming 기본 흐름 학습
- index.html 응답 수행

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: requesthandler는 코드 흐름에 대해서만 담당하고 http 파싱, 검증, 응답 반환은 별도 클래스로 진행해야 한다고 판단했음. 또한 파싱의 경우 abstract class로 공통 메소드(헤더 파싱)를 정의하고 body 파싱은 추상 메소드로 남겨둠.
- 그렇게 판단한 이유: requesthandler가 이 부분까지 담당하게 된다면 기능이 비대해진다고 생각했기 때문이고 http 파싱의 경우 헤더에 대한 파싱은 별반 차이가 없지만 body에 대한 파싱은 form이나 json, 혹은 없는 경우 등 content-type에 따라 달라지는 것을 고려했을 때 다형성을 활용하기 좋은 지점이라고 생각했기 때문임.
- 고려했지만 선택하지 않은 대안: 단순 if-else문, 인터페이스를 통해 확장성을 확보하는 것
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: socket 연결 없이 테스트가 가능해짐. 확장성이 올라감.

###### 3. 가장 막혔던 문제와 해결 과정
- http 파싱, 검증, 응답 반환의 경우 매개변수로만 동작하므로 utility 성격의 static 메소드로 만들지, 일반 클래스로 만들지 고민이 됐었음.
- 상태 변수가 없다면 그냥 static을 쓰는 게 나을 것이라고 생각했었음.
- 다만 http 버전에 따른 동작 차이나 미디어 타입에 따라 body 부분 파싱이 달라질 수 있다는 점에 대해서 ai 질문을 통해 알게 되었음.

###### 4. 오늘 아쉬웠던 행동(개발과 관련된 것)
- ai 의존성이 너무 높고 정답을 ai에서 찾으려고 함.
- java를 까먹어서 최대한 많은 예제를 따라치며 클래스 사용법을 익혀야 하는데 미흡했음.
